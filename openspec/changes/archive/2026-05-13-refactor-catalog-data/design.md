## Context

目前專案內的產品與分類資料大多是硬編碼或依賴零散的 mock JSON，這使得要根據 Excel 型錄檔案更新資料變得困難。新需求要求從 `AZX產品型錄.xlsx` 讀取資料，並以特定規則呈現（無子分類則直達列表、同名產品合併規格）。

## Goals / Non-Goals

**Goals:**
- 提供一個腳本或機制將 XLSX 轉為具有統一介面的 JSON。
- 更新側邊欄與分類導覽，固定大類並重設子類別。
- 實作「無子類別時直達產品列表」的路由/渲染邏輯。
- 實作「同名產品合併為一」並在 `Product.tsx` 中顯示多重規格的邏輯。
- 當沒有產品圖片時，顯示「圖片待更新」的 Placeholder 處理。

**Non-Goals:**
- 不改變現有的 UI 佈景主題 (Theme)。
- 暫不實作後端 API，所有操作與查詢均基於匯出後的靜態 JSON 檔案。

## Decisions

- **XLSX 轉 JSON**：將實作一個簡單的 Node.js 腳本（或手動使用工具加上邏輯），讀取 `AZX產品型錄.xlsx`，過濾掉第 2 列（大類：15的範例），並將所有的欄位標題映射為對應的英文 Key（過程中需忽略「數量」與「序」欄位，不匯出至 JSON 中）。然後以產品名稱為 Key 進行 GroupBy，將所有欄位封裝進規格列表 (`specs`) 中，缺失欄位補齊 `""`。
- **無子類別直達邏輯**：在 `Category` 頁面判斷，若當前點選的大類其 `subcategories` 陣列長度為 0 或為 undefined，則原本顯示子分類卡片的區塊直接渲染為 `ProductList`。
- **圖片 Fallback**：在渲染圖片的 `<img>` 標籤中，若 `imgUrl` 為空，則替換為一個帶有「圖片待更新」文字的預設 `div` 或預設圖。

## Risks / Trade-offs

- [Risk] 全端以 JSON 檔案為資料庫可能在資料量極大時影響效能。
  - Mitigation: 目前產品型錄資料量仍在前端可負荷範圍，未來若升級可輕易將讀取 JSON 的介面替換為 `fetch` API。
