## Context

目前產品明細頁中的規格欄位（如「電池容量」、「配置」等）會因商品不同而有所差異。為了降低系統維護成本並統一使用者體驗，我們需要將這些規格名稱統一。此外，配合業務流程調整，部分購買與結帳相關的 UI 元件需要在產品明細頁中隱藏，以簡化頁面呈現。

## Goals / Non-Goals

**Goals:**
- 將產品明細頁的規格名稱欄位統一改為「規格名稱」。
- 將產品明細頁的規格詳細內容統一改為「規格明細」，並實作為帶有 100 字元限制的 Text 文字框。
- 提供 mock API 資料以展示規格選擇時的連動效果。
- 隱藏不必要的購買相關按鈕、連結與數量欄位。

**Non-Goals:**
- 不涉及真實後端 API 的修改或串接。
- 不修改產品列表頁或其他頁面的顯示邏輯。
- 不改變購物車與報價單的核心商業邏輯，僅在 UI 層進行隱藏。

## Decisions

- **統一欄位名稱與樣式**：修改 `Product.tsx`，將原有的「電池容量」等字眼透過多語系或直接修改替換為「規格名稱」，並確保其 HTML class `option-group` 不被更動。
- **實作規格明細文字框**：新增一個 TEXT area 或 input 欄位作為「規格明細」，並設定字數上限 100 字（可能搭配 `maxLength={100}`），同時套用現有系統的 theme 樣式，設定為 `readOnly` 以免使用者編輯。
- **Mock 資料與連動**：在組件內部或獨立的 mock 檔案中建立一份 JSON 結構，當「規格名稱」選擇改變時，透過 React state (`useState`, `useEffect`) 同步更新「規格明細」的內容。
- **隱藏 UI 元件**：直接在 `Product.tsx` 中將「加入購物車」、「加入報價單/詢價單」、「使用以下方式購買」、「更多付款方式」以及「數量」欄位移除，或者以條件渲染 (`false && <Component />`) 隱藏，方便未來若有需求可快速恢復。

## Risks / Trade-offs

- [Risk] 直接在前端 mock 資料可能導致與未來真實 API 資料結構不符。
  - Mitigation：將 mock 資料結構盡可能貼近現有 `productApi.ts` 的設計，並封裝在獨立的函式或檔案中，以便未來抽換。
- [Risk] 隱藏結帳按鈕可能影響原有的 E2E 測試或組件依賴。
  - Mitigation：確認是否有測試依賴這些按鈕，如果有則一併調整測試；使用註解或配置來隱藏而非直接刪除程式碼。
